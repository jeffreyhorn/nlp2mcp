# ampl: Dollar condition set membership test missing parentheses ($119)

**GitHub Issue:** [#1003](https://github.com/jeffreyhorn/nlp2mcp/issues/1003)
**Model:** ampl (GAMSlib SEQ=74)
**Status:** FIXED
**Error category:** `path_syntax_error` (Subcategory C-adjacent — $119)
**Severity:** Medium — model parses and translates but GAMS compilation fails ($119 errors)

---

## Problem Summary

The ampl MCP output emits dollar conditions of the form `expr$t(tl)` where `t(tl)` is a set membership test ("is element `tl` a member of set `t`?"). GAMS requires parentheses around the condition in expression context: `expr$(t(tl))`. Without parentheses, GAMS interprets `$` as expecting a numeric value and raises Error $119 "Number expected".

Additionally, the emitter was outputting `==` for equality comparison (from the IR's `Binary("==", ...)` nodes generated by `.first`/`.last` translations), but GAMS requires `=` for equality in expressions. This caused $119 errors in conditions like `$(ord(tl) == card(tl))`.

---

## Root Cause

Two issues in `src/emit/expr_to_gams.py`:

1. **Dollar condition parenthesization**: The `DollarConditional` case only parenthesized conditions that were `Binary`, `Unary`, or `DollarConditional` nodes. `SetMembershipTest` (like `t(tl)`) and other condition types were emitted bare, causing GAMS to misparse `$t(tl)`.

2. **`==` operator not valid in GAMS**: The IR uses `Binary("==", ...)` for equality comparisons (generated by `.first`/`.last` translations in the parser), but GAMS only accepts `=` for equality and `<>` for inequality. The emitter passed `==` through verbatim, causing $119 errors.

---

## Fix Applied

Two changes in `src/emit/expr_to_gams.py`:

1. **Always parenthesize dollar conditions**: Replaced the conditional parenthesization (`if isinstance(condition, (Binary, Unary, DollarConditional))`) with unconditional parenthesization. Since `$(...)` is always valid GAMS syntax, this is safe for all condition types and eliminates ambiguity.

2. **Map `==`/`!=` to GAMS operators**: Added operator mapping `{"==": "=", "!=": "<>"}` in the Binary emission code, so IR comparison operators are correctly emitted as their GAMS equivalents. Also added `==` and `!=` to the `PRECEDENCE` table with the same precedence as `=` and `<>`.

Files modified:
- `src/emit/expr_to_gams.py`: Always parenthesize dollar conditions; map `==`/`!=` to GAMS operators
- `tests/unit/emit/test_expr_to_gams.py`: Updated 11 tests to expect parenthesized dollar conditions

## Notes

- The `$(...)` parenthesization is always safe and avoids all ambiguity for dollar conditions
- The `==` → `=` mapping only affects the emitter output; the IR continues to use `==` internally
- The `.first`/`.last` translation in the parser (lines 5017/5020) generates `Binary("==", ...)`, which is now correctly emitted as `=`
